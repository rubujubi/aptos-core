# Audit Output Format (CI edition)

> Read by: the `audit-pr` subagent after analysis agents return, to structure the final PR output.
>
> In CI, findings are posted as **PR review comments** — one inline comment per finding
> (keyed on file:line) plus a single summary comment at the top of the PR. There is no
> persisted `data/` evidence store in this deployment; the GitHub PR conversation IS the
> evidence store.

---

## Audit Summary Comment (posted once per run)

Post as a single PR comment via the GitHub API. Template:

```markdown
## Claude Audit: PR #{number} — {title}

**Risk**: {tier} ({score}/1.0)
**Subsystems**: {subsystem_list with criticality}
**Files**: {N} changed, +{added}/-{removed} lines
**Analyzers**: {which ran}

### Risk Score Breakdown

| Factor | Value | Reasoning |
|--------|-------|-----------|
| subsystem_criticality | {0.0-1.0} | {highest subsystem} |
| security_surface | {0.0-1.0} | {which surfaces} |
| exposure | {0.0-1.0} | {who can reach it} |
| divergence_score | {0.0-1.0} | {how different from upstream, or `n/a` if skipped} |
| change_magnitude | {0.0-1.0} | {lines changed} |
| feature_flag_exposure | {0.0-1.0} | {flags touched} |
| **composite** | **{score}** | **{tier}** |
| overrides | {any applied} | {reason} |

---

### Findings by Severity

| Severity | Count |
|----------|-------|
| critical | {n} |
| major (high) | {n} |
| minor (medium) | {n} |
| nit (low/info) | {n} |

### Top 3 to Address

1. **[{sev}]** {title} — `{file}:{line}` — {one-line why}
2. ...
3. ...

### Verdict

**Recommendation**: {merge / merge with follow-ups / hold for investigation / block}

**Follow-up items** (if any):
1. {action item}
2. ...

---

_Generated by the `audit-pr` subagent. Individual findings are posted as inline review comments on the relevant lines._
```

## Inline Review Comments (one per finding)

Post via the PR review API (`pulls/{number}/comments`), anchored to the specific file and
line the finding references. Template for each comment body:

```markdown
**[{severity_tag}] {title}**

{description — what is wrong}

**Attack Path**: {step-by-step}

**Suggested fix**:
```{language}
{concrete code snippet or description}
```

**Confidence**: {0.0-1.0} — {why}
**Category**: {security category}
```

Severity tag mapping (human-facing, per task spec):

| Internal severity | Tag |
|-------------------|-----|
| critical          | `[critical]` |
| high              | `[major]` |
| medium            | `[minor]` |
| low / info        | `[nit]` |

### Rules for inline comments

- One comment per finding. Do NOT batch multiple findings into one comment.
- Anchor on the most specific `file:line` the finding cites. If a finding cites a range,
  anchor to the start line and reference the range in the body.
- Do NOT post inline comments for:
  - Files not in the PR diff.
  - Generated files, lockfiles, or anything under `target/`, `dist/`, `vendor/`.
  - `## Hypothesis:` blocks (these are not findings — include them in an appendix of the
    summary comment if useful, never as inline comments).
- `## Deployment Footgun:` blocks MAY be posted inline at `[nit]` severity if they're
  tied to a specific line, or folded into the summary comment otherwise.

## Severity Counting Rules

The finding counts in the summary comment count ONLY `## Finding:` blocks.
`## Deployment Footgun:` and `## Hypothesis:` blocks are listed separately (in an
appendix of the summary comment) and do NOT contribute to severity counts.
